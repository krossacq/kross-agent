# SOP 04: Internal vs Client-Facing Visibility

Version: 0.1
Owner: Tax Mogul Bureau
Applies to: Kross Agent for Firms

## Purpose

Define how Kross Agent separates internal preparer information from client-facing information.

## Core Rule

Kross Agent must never expose internal preparer notes, AI drafts, raw reasoning, private firm SOPs, or unapproved summaries to the client portal.

Client-facing content must be explicitly approved before it becomes visible to the client.

## Visibility Types

Use these visibility labels:

- `internal`: visible only to authorized firm users.
- `preparer_only`: visible to assigned preparer and authorized admins/owners.
- `owner_admin_only`: visible to firm owner/admin.
- `client_facing_draft`: draft intended for client, not visible yet.
- `client_visible`: approved and visible to the client.
- `platform_admin_support`: visible only during approved support access.

## Internal Content

Internal content includes:

- Preparer notes.
- Tax judgment notes.
- Risk flags.
- Refund/balance-due discussion before approval.
- AI draft summaries.
- AI recommended memory updates before approval.
- Raw transcripts unless intentionally shared.
- Firm SOPs.
- Pricing rules not meant for client view.
- Escalation notes.
- Support notes.
- Credentials or access notes, though secrets should not be stored here at all.

## Client-Facing Content

Client-facing content may include:

- Approved document requests.
- Approved next steps.
- Approved deadlines.
- Approved appointment/reminder details.
- Approved client-facing call summaries.
- Approved status updates.
- Client tasks assigned to the client.

## AI Draft Handling

AI-generated content must start as a draft.

Drafts must include:

- draft status
- source reference
- generated timestamp
- generated by Kross Agent
- approval status
- reviewer, when reviewed

AI drafts are not official memory until approved.

## Approval Workflow

1. Kross creates draft content.
2. Kross labels content by proposed visibility.
3. Preparer or authorized user reviews the draft.
4. Reviewer edits if needed.
5. Reviewer approves or rejects.
6. Approved client-facing content is made visible to the client.
7. Approved internal content is saved to internal memory/timeline.
8. Action is logged.

## Client Portal Rule

The client portal can show only records where:

- `firm_id` matches the client's firm.
- `client_id` matches the logged-in client.
- visibility is `client_visible`.
- status is approved/active.

## Kross Response Rule

When Kross is answering in a client-facing context, it must not use:

- internal preparer notes
- firm SOPs
- AI drafts
- private support notes
- raw reasoning
- unapproved memory updates

Kross may use approved client-facing summaries, document requests, deadlines, and next steps.

## Acceptance Criteria

This SOP is working when:

- Internal notes never appear in the client portal.
- AI drafts are not client-visible by default.
- Client-facing next steps require approval.
- Kross can explain why a note is internal or client-facing.
- Visibility changes are audited.
